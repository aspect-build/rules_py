# `uv-pyproject-cases`

Regression coverage for distinct package-metadata shapes that rules_py's
sdist-build path has to handle — setup.py-only layouts, partial `[project]`
tables, dynamic dep declarations, native vs anyarch wheels, etc. Each
subdirectory pins one real PyPI package via the umbrella `uv.lock` (forced
through sdist with `[tool.uv].no-binary-package`) and asserts that
`pep517_whl` / `build_helper.py` route it correctly.

## Adding a new case

1. Add the package to the umbrella `pyproject.toml` `dependencies =`, pinned
   at the version that exhibits the shape you care about, and list it in
   `[tool.uv].no-binary-package` so uv resolves it from sdist. Re-run
   `uv lock` from this directory.
2. Create a subdirectory `cases/uv-pyproject-cases/<package-name>/`.
3. Add a `BUILD.bazel` declaring a `py_test` that depends on
   `@pypi_uv_pyproject_cases//<package-name>`. The umbrella's
   `default_build_dependencies = ["build", "setuptools"]` (see
   `setup.MODULE.bazel`) already wires `python -m build` into the
   autogenerated build_tool.
4. Drop an `__test__.py` that imports the package and exercises enough
   behavior to prove the wheel built and is loadable. The fact that the
   sdist build path resolved at all is most of the signal.

## Existing cases

- `pyahocorasick/` — setup.py + setup.cfg, no pyproject.toml. Regression
  for `dbc4272`, which narrowed `build_helper.py` so setup.py-only sdists
  go through `python -m build --no-isolation` instead of `python setup.py
  bdist_wheel` directly (the latter blows up on legacy `setup_requires=`
  packages like googlemaps 4.10.0 via `fetch_build_eggs`).
