# Protobuf + rules_py interop example

This standalone module demonstrates — and tests — that `rules_py` targets can
consume Python code generated by [protobuf]'s `py_proto_library`.

## Why this is interesting

`py_proto_library` generates a `py_library` (carrying `PyInfo`) and pulls in the
protobuf Python runtime. A `rules_py` `py_binary`/`py_test` consumes that
`PyInfo` directly, so the generated bindings work inside a `rules_py` venv with
no glue code.

The only direct dependencies are `aspect_rules_py` and `protobuf`: nothing here
names [rules_python], even though both rulesets rely on it transitively, and the
Python interpreter toolchain is provisioned through `rules_py`'s own
`python_interpreters` extension. This example exists so that contract is built
and tested on CI.

## Layout

| File | Role |
| --- | --- |
| `greeting.proto` | A trivial `Greeting` message. |
| `BUILD.bazel` | `proto_library` → `py_proto_library` → `rules_py` `py_binary`/`py_test`. |
| `app.py` | Imports the generated `greeting_pb2`, builds and round-trips a message. |
| `app_test.py` | Asserts the generated module imports and serializes under a `rules_py` venv. |

## Try it

```sh
bazel run //:app
bazel test //:app_test
```

[protobuf]: https://github.com/protocolbuffers/protobuf
[rules_python]: https://github.com/bazelbuild/rules_python
